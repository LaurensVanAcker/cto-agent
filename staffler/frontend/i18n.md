# Internationalization

## Library: `@ngx-translate/core` v16

Beide repos gebruiken `ngx-translate` voor runtime-locale-switching. Geen Angular i18n (geen `$localize`, geen XLF). Reden: ngx-translate is dynamic, kan locale switchen zonder rebuild — handig voor Belgische klanten met NL+EN gebruikers door elkaar.

## dps (NL + EN)

### Configuratie

`src/app/core/i18n/i18n.module.ts`:
```ts
@NgModule({
  providers: [
    { provide: LOCALE_ID, useValue: AppLocaleEnum.NL },
    { provide: DEFAULT_CURRENCY_CODE, useValue: 'EUR' },
    provideHttpClient(withInterceptorsFromDi()),
    provideTranslateService({
      loader: { provide: TranslateLoader, useFactory: HttpLoaderFactory, deps: [HttpClient] },
    }),
  ],
})
export class I18nModule {
  constructor(private translateService: TranslateService, private primeConfig: PrimeNG) {
    registerLocaleData(localeNl, AppLocaleEnum.NL);
    this.translateService.langs.push(...Object.values(AppLocaleEnum));

    let defaultLocale = localStorage.getItem(APP_LOCALE_KEY);
    if (!defaultLocale || !this.translateService.getLangs().includes(defaultLocale)) {
      defaultLocale = environment.envName === EnvNameEnum.DEV ? AppLocaleEnum.EN : AppLocaleEnum.NL;
    }
    this.translateService.setDefaultLang(defaultLocale);
    this.translateService.use(this.translateService.defaultLang);

    this.translateService.onLangChange.asObservable().subscribe(({ lang, translations }) => {
      document.documentElement.lang = lang;
      localStorage.setItem(APP_LOCALE_KEY, lang);
      this.primeConfig.setTranslation(translations['primeng']);
      Settings.defaultLocale = lang;
      LocaleManager.applyLocale(APP_TO_SCHEDULER_LOCALES_MAP[lang as AppLocaleEnum]);
    });
  }
}
```

Opmerkingen:
- **`LOCALE_ID = 'nl'`** als hardcoded fallback (Belgian Dutch via `@angular/common/locales/nl-BE`).
- **`DEFAULT_CURRENCY_CODE = 'EUR'`** voor `currency` pipe.
- **Default locale** komt uit `localStorage.dpsLocale` als die set is, anders:
  - `EN` op DEV (developer convenience)
  - `NL` op QA en PROD (klanten verwachten Nederlands)
- Op locale-change worden 4 dingen tegelijk aangepast:
  1. `<html lang="...">` (voor SEO + screen-readers)
  2. `localStorage.dpsLocale`
  3. PrimeNG translations (datepicker labels, etc.) via `primeConfig.setTranslation(translations['primeng'])`
  4. Luxon `Settings.defaultLocale` (date formatting)
  5. Bryntum Scheduler locale (`LocaleManager.applyLocale('Nl' | 'En')`)

### `AppLocaleEnum`

```ts
export enum AppLocaleEnum {
  NL = 'nl',
  EN = 'en',
}
```

Geen sub-locales (`nl-BE` of `en-US`). De PrimeNG- en Luxon-conventies worden gevoed door de app-default `nl-BE` registration.

### Translation files

`src/assets/i18n/`:
- `en.json` (36593 bytes)
- `nl.json` (38208 bytes)
- `scheduler.locale.En.js` (428 bytes) — Bryntum English locale
- `scheduler.locale.Nl.js` (429 bytes) — Bryntum Dutch locale

JSON-formaat (nested object), geladen via `TranslateHttpLoader` op `/assets/i18n/en.json` en `/assets/i18n/nl.json` (default loader path).

### Gebruik in templates

```html
<p-button [label]="'AUTH.LOGIN' | translate"></p-button>
<input [placeholder]="'AUTH.EMAIL_LABEL' | translate" />
```

In code:
```ts
this.translateService.instant('AUTH.LOGIN_ERROR')                  // sync
this.translateService.stream('AUTH.LOGIN').subscribe(...)          // observable stream
this.translateService.use('en')                                    // switch locale
```

### Locale-switcher

`AppLocaleSelectorComponent` in `shared/components/app-locale-selector/`. Toont een dropdown NL/EN, op selectie roept `translateService.use(...)` aan. Persistent via de `onLangChange` subscribe in `I18nModule`.

### PrimeNG translations key

Conventie: in `nl.json` en `en.json` staat een top-level key `"primeng": { ... }` met alle PrimeNG-component labels (firstDayOfWeek, dayNames, monthNames, choose, upload, cancel, ...). `i18n.module.ts` extract die object en geeft door aan `primeConfig.setTranslation(...)`.

Dat is een Praktische standaard: alle vendor-translations in één JSON, geen aparte vendor-files inladen.

### Bryntum Scheduler locales

Bryntum heeft eigen locale-files in JS-format (niet JSON). Deze worden static geïmporteerd in `i18n.module.ts`:

```ts
import '../../../assets/i18n/scheduler.locale.En.js';
import '../../../assets/i18n/scheduler.locale.Nl.js';
```

Dat registreert ze globaal in `LocaleManager`. Op locale-change roept de subscribe `LocaleManager.applyLocale('En' | 'Nl')` (let op caps).

## my-staffler (NL + EN, EN default)

### Configuratie

`src/app/core/i18n/http-loader.factory.ts`:
```ts
export function HttpLoaderFactory(http: HttpClient): TranslateHttpLoader {
  return new TranslateHttpLoader(http, './assets/i18n/', '.json');
}
```

`src/app/app.config.ts`:
```ts
importProvidersFrom(
  TranslateModule.forRoot({
    loader: { provide: TranslateLoader, useFactory: HttpLoaderFactory, deps: [HttpClient] },
    defaultLanguage: 'en',
  }),
  ...
)
```

Verschillen met dps:
- **`defaultLanguage: 'en'`** in alle envs (geen NL fallback). Daar is enkel een `assets/i18n/en.json` aanwezig (voor zover gevonden). Voor mobile-werknemers in BE is dit een tekortkoming (zie `quirks.md`).
- Geen LOCALE_ID setting → blijft default `en-US`.
- Geen Luxon (gebruikt `date-fns` in plaats), dus geen `Settings.defaultLocale` toelichting.
- Geen PrimeNG → geen translation-injection in vendor.
- Ionic-component labels worden niet vertaald in code (Ionic gebruikt zelf system-locale waar mogelijk).

### Translation files

`src/assets/i18n/` — alleen één folder gevonden, `en.json` zou daar moeten zitten (file niet expliciet gelezen). Geen `nl.json` in de root listing.

Indien Belgische werknemers wél NL moeten krijgen, is dit een direct gat. De app gaat in iOS/Android default in NL als de OS-locale NL is, maar `defaultLanguage: 'en'` overrules dat in ngx-translate.

### Gebruik in templates

```html
<ion-label>{{ 'tabs.schedule' | translate }}</ion-label>
<ion-label>{{ 'tabs.clockIn' | translate }}</ion-label>
<ion-label>{{ 'tabs.profile' | translate }}</ion-label>
```

Translation keys gebruiken **lowercase camelCase / dot.notation**: `tabs.schedule`, `auth.login.error`. Dps gebruikt **UPPERCASE.SNAKE_CASE**: `AUTH.LOGIN`, `AUTH.EMAIL_LABEL`. Inconsistent tussen de twee repos.

### Locale-switcher

Geen UI-control gevonden in mystaffler. Geen `LocaleSelectorComponent`. De user kan dus de language niet via de app aanpassen, alleen via OS-instellingen (en dan alleen als ngx-translate ooit een fallback-naar-system implementeert, wat het niet doet).

## Date/time formatting

| Aspect | dps | mystaffler |
|---|---|---|
| Library | Luxon 3.4 | date-fns 4 |
| Default locale | gevolgd via `Settings.defaultLocale` | n.v.t. — date-fns is locale-explicit-per-call |
| Belgian dates | `18/11/2026` | onbekend |
| Belgian times | `14u30` | onbekend |
| Currency | `EUR`, format via `currency` pipe | onbekend (geen pipe) |

`FormatDatetimePipe` in dps formats Luxon DateTime naar Belgian-style strings.

## Cross-language fallbacks

ngx-translate heeft een `defaultLang` mechanisme:
- Als de huidige `lang` een key niet heeft, valt het terug op de `defaultLang` JSON.
- Als de `defaultLang` JSON ook niet heeft, returnt `translateService.instant(...)` de key zelf (bv. `'AUTH.LOGIN'`).

Voor de PoC: hou de NL en EN files in sync. CI heeft geen lint die ontbrekende keys detecteert.

dps:

```
AUTH.EMAIL_LABEL

```

mystaffler:
```

tabs.clockIn
tabs.profile
auth.login.error
...
```

Wenselijk voor convergentie: dezelfde naming convention. Aanbeveling: `feature.subFeature.key` lowercase camelCase. Past bij JSON-natuur en is consistent met JS field-naming.

## Pluralization

Geen ICU MessageFormat in beide repos. ngx-translate ondersteunt het via `@ngx-translate/messageformat-compiler` (niet geïnstalleerd). Pluralization wordt vandaag opgelost door **specifieke keys per count**:

```json
{
  "RESULTS_COUNT.ZERO": "Geen resultaten",
  "RESULTS_COUNT.ONE": "1 resultaat",
  "RESULTS_COUNT.MANY": "{{ count }} resultaten"
}
```

En component-side een conditionele key-keuze. Brittle maar werkt.

## Wat ontbreekt + advies

1. **mystaffler heeft geen NL** vandaag. Voor de PoC die op klanten getoond wordt: voeg `nl.json` toe en zet `defaultLanguage: 'nl'`.
2. **Geen ICU pluralization** — als je >5 verschillende plural-cases hebt, overweeg `@ngx-translate/messageformat-compiler` toe te voegen.
3. **Geen lint voor ontbrekende keys** — een kleine CI-check (script die `nl.json` keys vergelijkt met `en.json` keys) zou drift voorkomen.
4. **Geen i18n-extract** — de keys worden manueel beheerd. ngx-translate heeft géén "extract from templates" tool zoals Angular i18n. Voor de PoC: hou een lijst bij in een README.
5. Voor een gedeelde `@staffler/i18n` lib in een toekomstige monorepo: bundel beide JSONs en biedt één locale-switch service.
